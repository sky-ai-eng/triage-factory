GitHub interactions use `{{BINARY_PATH}} exec gh`. Run `{{BINARY_PATH}} exec gh --help` for the full command list, and `--help` on any subcommand for details.

Workflows:
- Inspect a PR: `exec gh pr view`, `exec gh pr diff`, `exec gh pr files`. Use `-v` for full review/comment bodies.
- Review a PR: `exec gh pr start-review` → `exec gh pr add-review-comment` (repeat) → `exec gh pr submit-review`.
- Investigate a failing CI run: `exec gh actions download-logs <workflow_run_id>` downloads and extracts the full log archive to `./_scratch/ci-logs/<run_id>/`. Returns JSON with `run_id`, `owner`, `repo`, `dest_dir`, `bytes_downloaded`, and an `entries` array listing the top-level contents. Use your standard Read/Grep/Glob tools on the extracted files — each top-level directory under the destination is a single job.
- Read files in the working directory for context beyond the diff.

Repo target resolution:
- All gh commands target a (owner, repo) pair. Resolution order: an explicit `--repo owner/repo` flag, then the `TODOTRIAGE_REPO` environment variable (set automatically for delegated runs with a resolved repo context), then the current directory's `.git/config` origin.
- You almost never need to pass `--repo` — the env var is already set for you when your run has a repo context.

Gotchas:
- The `--body` for `submit-review` should begin with a `##` title (e.g. "## Looks good — minor nits") followed by your summary. A top-level heading and metadata footer are added automatically; use `##` for yours.
- For multi-line `--body` arguments, use a heredoc:
  ```
  {{BINARY_PATH}} exec gh pr add-review-comment <review_id> --file <path> --line <N> --body "$(cat <<'EOF'
  Your comment here.
  Can span multiple lines, include "quotes", and code blocks.
  EOF
  )"
  ```
